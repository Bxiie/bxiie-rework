Bxiie CMS Git Deployment Cookbook
===================================

Purpose
-------

This cookbook walks through setting up the Bxiie CMS source repository on the development workstation and deploying it to the production Apache2 server using GitHub.

The deployment model is:

- Development workstation owns the editable Git repository.
- GitHub becomes the canonical origin repository.
- Production server pulls from GitHub.
- Production deployment is performed by a deploy script or Git pull.
- Apache serves only /var/www/bxiie-cms/public.
- Persistent runtime data stays on the server and is not overwritten by Git:
  - SQLite database
  - uploaded originals
  - generated image cache
  - private files
  - environment configuration


Important Local Repository Path
-------------------------------

The local Git working repository path is:

/Users/bxiie/Dropbox/artsy/site/bxiie_rework

This cookbook assumes:

- macOS development workstation
- Dropbox-backed working tree
- Git operations occur from:

/Users/bxiie/Dropbox/artsy/site/bxiie_rework

Be aware that Dropbox syncing and Git do not always behave perfectly together during large binary/image operations.

Practical recommendation:

- let Git complete operations before Dropbox fully syncs
- avoid editing the same repo from multiple machines simultaneously
- avoid committing partially synced binary files

Part 1: Initialize the Local Repository
---------------------------------------

1. Create the directory:

    mkdir -p /Users/bxiie/Dropbox/artsy/site
    cd /Users/bxiie/Dropbox/artsy/site

2. Unpack the scaffold:

    cd /Users/bxiie/Dropbox/artsy/site
    unzip bxiie_cms_scaffold.zip
    mv bxiie_cms_scaffold bxiie_rework
    cd bxiie_rework

Result:

/Users/bxiie/Dropbox/artsy/site/bxiie_rework


3. Create .gitignore:

Target file:

/Users/bxiie/Dropbox/artsy/site/bxiie_rework/.gitignore

Contents:

    # Local environment files.
    .env
    .env.*
    !.env.example

    # SQLite databases.
    database/*.sqlite
    database/*.sqlite-*
    database/*.db
    database/*.db-*

    # Runtime upload/cache storage.
    storage/uploads/*
    storage/cache/*
    storage/private/*

    # Keep storage directories.
    !storage/uploads/.gitkeep
    !storage/cache/.gitkeep
    !storage/private/.gitkeep

    # macOS junk.
    .DS_Store

    # IDE/editor junk.
    .idea/
    .vscode/
    *.swp
    *.swo

    # Logs.
    *.log

    # End of file.

Create keep files:

    touch storage/uploads/.gitkeep
    touch storage/cache/.gitkeep
    touch storage/private/.gitkeep


4. Initialize Git:

    cd /Users/bxiie/Dropbox/artsy/site/bxiie_rework

    git init
    git add .
    git commit -m "Initial Bxiie CMS scaffold"
    git branch -M main

Verify:

    git status

Expected:

    nothing to commit, working tree clean


Part 2: Create the GitHub Repository
------------------------------------

1. Create the GitHub repository.

Create a new private repository on GitHub.

Suggested name:

bxiie-rework

Do not initialize it with:

- README
- .gitignore
- license

because the local repository already exists.


2. Add GitHub as origin using GH_TOKEN.

This guide uses GH_TOKEN for GitHub authentication.

Use a GitHub fine-grained personal access token with access to the repository.

Minimum practical permissions:

- Contents: Read and write
- Metadata: Read-only

Do not commit the token.

Set it in the current shell only:

    export GH_TOKEN='paste-your-github-token-here'

From:

/Users/bxiie/Dropbox/artsy/site/bxiie_rework

run:

    cd /Users/bxiie/Dropbox/artsy/site/bxiie_rework

    git remote add origin https://x-access-token:${GH_TOKEN}@github.com/YOUR_GITHUB_USERNAME/bxiie-rework.git

Verify:

    git remote -v

Expected:

    origin https://x-access-token:GH_TOKEN_VALUE@github.com/YOUR_GITHUB_USERNAME/bxiie-rework.git (fetch)
    origin https://x-access-token:GH_TOKEN_VALUE@github.com/YOUR_GITHUB_USERNAME/bxiie-rework.git (push)

Trap: git remote -v may display the token. Do not paste that output into tickets, chats, docs, screenshots, or commit messages.

Safer alternative: store a token-free remote URL and use macOS Keychain or another Git credential helper. For this project, the explicit GH_TOKEN pattern is acceptable if handled carefully.


3. Push initial source to GitHub:

    cd /Users/bxiie/Dropbox/artsy/site/bxiie_rework

    export GH_TOKEN='paste-your-github-token-here'

    git push -u origin main

GitHub now becomes the canonical upstream repository.

The workstation is no longer the source of truth.

The GitHub repository is.


Part 3: Configure the Production Server
---------------------------------------

1. Install Git and required packages:

    sudo apt update
    sudo apt install -y git unzip apache2 php php-cli php-sqlite3 php-gd php-mbstring libapache2-mod-php
    sudo a2enmod rewrite headers
    sudo systemctl restart apache2


2. Create application directories:

    sudo mkdir -p /var/www/bxiie-cms
    sudo mkdir -p /var/lib/bxiie-cms/database
    sudo mkdir -p /var/lib/bxiie-cms/storage/uploads
    sudo mkdir -p /var/lib/bxiie-cms/storage/cache
    sudo mkdir -p /var/lib/bxiie-cms/storage/private
    sudo mkdir -p /etc/bxiie-cms
    sudo mkdir -p /var/backups/bxiie-cms


3. Clone the GitHub repository onto production using GH_TOKEN.

Create a production GitHub token environment file:

/etc/bxiie-cms/github.env

    sudo tee /etc/bxiie-cms/github.env > /dev/null <<'EOF'
    GH_TOKEN=paste-your-github-token-here
    GITHUB_OWNER=YOUR_GITHUB_USERNAME
    GITHUB_REPO=bxiie-rework
    EOF

    sudo chown root:root /etc/bxiie-cms/github.env
    sudo chmod 600 /etc/bxiie-cms/github.env

Clone the repository:

    sudo bash -c '
    set -euo pipefail
    source /etc/bxiie-cms/github.env
    cd /var/www
    git clone "https://x-access-token:${GH_TOKEN}@github.com/${GITHUB_OWNER}/${GITHUB_REPO}.git" bxiie-cms
    '

Immediately remove the token from the stored Git remote:

    sudo bash -c '
    set -euo pipefail
    source /etc/bxiie-cms/github.env
    cd /var/www/bxiie-cms
    git remote set-url origin "https://github.com/${GITHUB_OWNER}/${GITHUB_REPO}.git"
    '

Trap: cloning with the token in the URL works, but Git may persist the full tokenized remote URL. Always reset the remote URL immediately after cloning.


4. Configure production Git pulls with GH_TOKEN.

Production pulls should inject GH_TOKEN at command time instead of storing it in the Git remote.

Create:

/usr/local/bin/bxiie-git-pull

    sudo tee /usr/local/bin/bxiie-git-pull > /dev/null <<'EOF'
    #!/bin/bash
    # Pull Bxiie CMS source from GitHub using GH_TOKEN without storing the token
    # in the repository remote URL.

    set -euo pipefail

    ENV_FILE="/etc/bxiie-cms/github.env"
    APP_DIR="/var/www/bxiie-cms"
    BRANCH="main"

    if [[ ! -r "$ENV_FILE" ]]; then
      echo "Missing readable GitHub environment file: $ENV_FILE" >&2
      exit 1
    fi

    source "$ENV_FILE"

    if [[ -z "${GH_TOKEN:-}" || -z "${GITHUB_OWNER:-}" || -z "${GITHUB_REPO:-}" ]]; then
      echo "GH_TOKEN, GITHUB_OWNER, and GITHUB_REPO must be set in $ENV_FILE" >&2
      exit 1
    fi

    cd "$APP_DIR"

    git remote set-url origin "https://github.com/${GITHUB_OWNER}/${GITHUB_REPO}.git"

    git -c http.extraHeader="Authorization: Bearer ${GH_TOKEN}" fetch origin "$BRANCH"
    git checkout "$BRANCH"
    git reset --hard "origin/${BRANCH}"

    # End of file.
    EOF

    sudo chown root:root /usr/local/bin/bxiie-git-pull
    sudo chmod 750 /usr/local/bin/bxiie-git-pull

Verify:

    sudo /usr/local/bin/bxiie-git-pull


5. Set ownership:

    sudo chown -R www-data:www-data /var/lib/bxiie-cms
    sudo chown -R deploy:www-data /var/www/bxiie-cms
    sudo chmod -R 2775 /var/www/bxiie-cms
    sudo chmod -R 2775 /var/lib/bxiie-cms


6. Update production.

To deploy future changes:

    sudo /usr/local/bin/bxiie-git-pull
    sudo apachectl configtest
    sudo systemctl reload apache2

Recommended future improvement:

Create:

/usr/local/bin/bxiie-deploy

which wraps:

- /usr/local/bin/bxiie-git-pull
- database backup
- PHP syntax checks
- migrations
- cache cleanup
- Apache config test
- Apache reload

That becomes the official deployment entry point instead of raw Git commands.


Part 4: Daily Development Workflow
----------------------------------

1. Pull latest state if developing from multiple workstations:

    cd /Users/bxiie/Dropbox/artsy/site/bxiie_rework

    git pull


2. Make changes.

Edit files normally.


3. Check PHP syntax:

    cd /Users/bxiie/Dropbox/artsy/site/bxiie_rework

    find . -path './.git' -prune -o -name '*.php' -print0 | xargs -0 -n1 php -l

Expected:

    No syntax errors detected


4. Commit changes:

    cd /Users/bxiie/Dropbox/artsy/site/bxiie_rework

    git status
    git add .
    git commit -m "Describe the change"


5. Deploy.

Step 1: Push changes to GitHub using GH_TOKEN:

    cd /Users/bxiie/Dropbox/artsy/site/bxiie_rework

    export GH_TOKEN='paste-your-github-token-here'

    git push origin main

Step 2: Pull changes onto production:

    ssh deploy@bxiie.com

    sudo /usr/local/bin/bxiie-git-pull
    sudo apachectl configtest
    sudo systemctl reload apache2


6. Verify production:

    curl -I https://bxiie.com/
    curl -I https://bxiie.com/admin/login

Expected:

    HTTP/1.1 200 OK

Strong recommendation regarding uploads
---------------------------------------

Do NOT commit uploaded artwork originals into Git.

Use Git for:

- source
- templates
- CSS
- JS
- configs
- migrations
- docs

Use persistent runtime storage for:

- uploaded images
- generated thumbnails
- caches
- exports
- backups


Strong recommendation regarding image derivatives
-------------------------------------------------

Generated derivatives should eventually live under:

/var/lib/bxiie-cms/storage/cache

and be regenerated automatically.

Never hand-edit generated derivatives.


Strong recommendation regarding multitenancy
--------------------------------------------

From the beginning:

- every table should have tenant_id
- every image belongs to a tenant
- every admin belongs to one or more tenants
- every stats query filters by tenant
- every uploaded file path should partition by tenant slug or ID
