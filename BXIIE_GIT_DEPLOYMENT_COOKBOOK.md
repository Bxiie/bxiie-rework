# Bxiie CMS Git Deployment Cookbook

## Purpose

This cookbook walks through setting up the Bxiie CMS source repository on a development workstation and deploying it to an existing Apache2/Certbot production server using Git.

The deployment model is:

- Development workstation owns the editable Git repository.
- Production server owns a bare Git repository used as a deploy target.
- A Git `post-receive` hook checks out the deployed branch into `/var/www/bxiie-cms`.
- Apache serves only `/var/www/bxiie-cms/public`.
- Persistent runtime data stays on the server and is not overwritten by Git:
  - SQLite database
  - uploaded originals
  - generated image cache
  - private files
  - environment configuration

## Confirmed / Assumed / Unknown

### Confirmed

- Current Bxiie site is static HTML.
- New application is a PHP/SQLite custom CMS scaffold.
- Apache2 and Certbot already exist on the production server.
- The CMS config supports:
  - `DATABASE_PATH`
  - `STORAGE_PATH`
  - `BASE_URL`
  - `DEFAULT_DOMAIN`
  - `DEFAULT_TENANT_SLUG`
  - `ADMIN_PATH`
- Apache must serve the app from the `public/` directory only.

### Assumed

- Production server is Ubuntu or Debian.
- Apache service name is `apache2`.
- Apache runtime user is `www-data`.
- Production app path will be `/var/www/bxiie-cms`.
- Production persistent data path will be `/var/lib/bxiie-cms`.
- Production environment file path will be `/etc/bxiie-cms/bxiie-cms.env`.
- Bare Git repository path will be `/srv/git/bxiie-cms.git`.
- Deploy branch will be `main`.
- The SSH user used for deployment is `deploy`.

### Unknown

- Exact existing Apache vhost filename.
- Exact current document root for `bxiie.com`.
- Whether PHP, `pdo_sqlite`, and `gd` are already installed.
- Whether the current server uses Apache PHP module or PHP-FPM.

## Important traps

### Do not point Apache at the repository root

Apache must point here:

```text
/var/www/bxiie-cms/public
```

Never point Apache here:

```text
/var/www/bxiie-cms
```

The app root contains config, database, app code, scripts, and storage paths that should not be web-accessible.

### Do not keep production uploads only inside Git checkout

The Git checkout can be replaced during deployment. Uploaded images and the SQLite database must live outside Git-controlled source or be symlinked into the application.

This cookbook uses:

```text
/var/lib/bxiie-cms/database/bxiie.sqlite
/var/lib/bxiie-cms/storage
```

### Do not rerun Certbot unless certificate domains change

You already have Certbot configuration. Preserve existing certificate lines and only update the document root for the Bxiie vhost.

### Do not commit secrets

Do not commit production passwords, private keys, mail credentials, or `.env` files.

## Target directory layout

Production server:

```text
/srv/git/bxiie-cms.git                 # Bare Git deployment repository
/var/www/bxiie-cms                     # Checked-out application source
/var/lib/bxiie-cms/database            # Persistent SQLite database directory
/var/lib/bxiie-cms/storage             # Persistent uploads/cache/private storage
/etc/bxiie-cms/bxiie-cms.env           # Production environment variables
/var/backups/bxiie-cms                 # Deployment and database backups
```

Development workstation:

```text
~/src/bxiie-cms
```

Adjust these paths if your local convention differs.

# Part 1: Set up the development workstation repository

## 1. Create a working directory

On your development workstation:

```bash
mkdir -p ~/src
cd ~/src
```

## 2. Unpack the scaffold

Put `bxiie_cms_scaffold.zip` in `~/src`, then run:

```bash
cd ~/src
unzip bxiie_cms_scaffold.zip
mv bxiie_cms_scaffold bxiie-cms
cd bxiie-cms
```

## 3. Create `.gitignore`

Create this file:

```bash
cat > .gitignore <<'EOF'
# Local environment files.
.env
.env.*
!.env.example

# Runtime database files.
database/*.sqlite
database/*.sqlite-*
database/*.db
database/*.db-*

# Runtime storage generated by uploads and cache.
storage/uploads/*
storage/cache/*
storage/private/*

# Keep storage directories in the repo.
!storage/uploads/.gitkeep
!storage/cache/.gitkeep
!storage/private/.gitkeep

# Local OS/editor junk.
.DS_Store
.idea/
.vscode/
*.swp
*.swo

# Logs and temporary files.
*.log
tmp/
cache/

# End of file.
EOF
```

Create `.gitkeep` files so the empty runtime directories exist in Git:

```bash
touch storage/uploads/.gitkeep
touch storage/cache/.gitkeep
touch storage/private/.gitkeep
```

## 4. Create a local environment example

Create this file:

```bash
cat > .env.example <<'EOF'
# Example local environment for Bxiie CMS.
# Copy this file to .env for local development and adjust values.

APP_NAME="Bxiie Artist CMS"
BASE_URL="http://localhost:8080"
DATABASE_PATH="./database/bxiie.sqlite"
STORAGE_PATH="./storage"
ADMIN_PATH="/admin"
SESSION_NAME="bxiie_admin"
DEFAULT_TENANT_SLUG="bxiie"
DEFAULT_DOMAIN="localhost"
WATERMARK_TEXT="© Bxiie"
MAIL_FROM="noreply@bxiie.com"
CONTACT_TO="artist@bxiie.com"

# End of file.
EOF
```

The current scaffold reads environment variables directly through PHP `getenv()`. It does not automatically parse `.env`; `.env.example` is documentation until a loader is added.

## 5. Initialize Git

```bash
git init
git add .
git commit -m "Initial Bxiie CMS scaffold"
git branch -M main
```

## 6. Optional: run local PHP syntax checks

From the repo root:

```bash
find . -path './.git' -prune -o -name '*.php' -print0 | xargs -0 -n1 php -l
```

Expected result: every PHP file reports `No syntax errors detected`.

# Part 2: Prepare the production server

## 1. Create a deployment user

On the production server:

```bash
sudo adduser --disabled-password --gecos "Bxiie deploy user" deploy
sudo usermod -aG www-data deploy
```

Add your workstation SSH public key:

```bash
sudo mkdir -p /home/deploy/.ssh
sudo nano /home/deploy/.ssh/authorized_keys
sudo chown -R deploy:deploy /home/deploy/.ssh
sudo chmod 700 /home/deploy/.ssh
sudo chmod 600 /home/deploy/.ssh/authorized_keys
```

Verify from your workstation:

```bash
ssh deploy@bxiie.com 'whoami && hostname'
```

Expected output includes:

```text
deploy
```

## 2. Install required packages

On the production server:

```bash
sudo apt update
sudo apt install -y git unzip apache2 php php-cli php-sqlite3 php-gd php-mbstring libapache2-mod-php
sudo a2enmod rewrite headers
sudo systemctl restart apache2
```

If your server uses PHP-FPM instead of `libapache2-mod-php`, do not blindly install both models on a tuned server. Check first:

```bash
apachectl -M | grep -E 'php|proxy_fcgi'
systemctl list-units 'php*-fpm.service'
```

## 3. Create production directories

```bash
sudo mkdir -p /srv/git
sudo mkdir -p /var/www/bxiie-cms
sudo mkdir -p /var/lib/bxiie-cms/database
sudo mkdir -p /var/lib/bxiie-cms/storage/uploads
sudo mkdir -p /var/lib/bxiie-cms/storage/cache
sudo mkdir -p /var/lib/bxiie-cms/storage/private
sudo mkdir -p /etc/bxiie-cms
sudo mkdir -p /var/backups/bxiie-cms

sudo chown -R deploy:www-data /var/www/bxiie-cms
sudo chown -R www-data:www-data /var/lib/bxiie-cms
sudo chown -R deploy:deploy /srv/git
sudo chmod -R 2775 /var/www/bxiie-cms
sudo chmod -R 2775 /var/lib/bxiie-cms
sudo chmod 750 /etc/bxiie-cms
sudo chmod 750 /var/backups/bxiie-cms
```

## 4. Create the bare Git repository

```bash
sudo -u deploy git init --bare /srv/git/bxiie-cms.git
```

Create `/etc/bxiie-cms/bxiie-cms.env`:

```bash
sudo tee /etc/bxiie-cms/bxiie-cms.env > /dev/null <<'EOF'
```

# Part 3: Create the Git post-receive deploy hook

Create `/srv/git/bxiie-cms.git/hooks/post-receive`:

```bash
sudo -u deploy tee /srv/git/bxiie-cms.git/hooks/post-receive > /dev/null <<'EOF'
#!/bin/bash
#
# Deploy Bxiie CMS after a successful push to the bare production repository.
# This hook checks out the main branch to /var/www/bxiie-cms and preserves
# runtime database/storage under /var/lib/bxiie-cms.
#
# End of file marker is intentionally kept as a shell comment at the bottom.

set -euo pipefail

DEPLOY_BRANCH="main"
BARE_REPO="/srv/git/bxiie-cms.git"
WORK_TREE="/var/www/bxiie-cms"
BACKUP_DIR="/var/backups/bxiie-cms"
PHP_BIN="/usr/bin/php"
APACHE_SERVICE="apache2"

log() {
  printf '[bxiie deploy] %s\n' "$1"
}

while read -r oldrev newrev refname; do
  branch="${refname#refs/heads/}"

  if [[ "$branch" != "$DEPLOY_BRANCH" ]]; then
    log "Ignoring push to branch ${branch}; deploy branch is ${DEPLOY_BRANCH}."
    continue
  fi

  log "Starting deployment of ${DEPLOY_BRANCH}."

  if [[ ! -d "$WORK_TREE" ]]; then
    log "Creating work tree ${WORK_TREE}."
    mkdir -p "$WORK_TREE"
  fi

  if [[ -f "/var/lib/bxiie-cms/database/bxiie.sqlite" ]]; then
    backup_file="${BACKUP_DIR}/bxiie.sqlite.$(date +%Y%m%d-%H%M%S).bak"
    log "Backing up SQLite database to ${backup_file}."
    cp "/var/lib/bxiie-cms/database/bxiie.sqlite" "$backup_file"
  fi

  log "Checking out source into ${WORK_TREE}."
  git --git-dir="$BARE_REPO" --work-tree="$WORK_TREE" checkout -f "$DEPLOY_BRANCH"

  log "Ensuring runtime directories exist."
  mkdir -p /var/lib/bxiie-cms/database
  mkdir -p /var/lib/bxiie-cms/storage/uploads
  mkdir -p /var/lib/bxiie-cms/storage/cache
  mkdir -p /var/lib/bxiie-cms/storage/private

  log "Checking PHP syntax."
  find "$WORK_TREE" -path "$WORK_TREE/.git" -prune -o -name '*.php' -print0 | xargs -0 -n1 "$PHP_BIN" -l >/tmp/bxiie-php-lint.log
  cat /tmp/bxiie-php-lint.log

  log "Setting ownership and permissions."
  chown -R deploy:www-data "$WORK_TREE"
  chmod -R g+rwX "$WORK_TREE"
  find "$WORK_TREE" -type d -exec chmod g+s {} \;

  chown -R www-data:www-data /var/lib/bxiie-cms
  chmod -R 2775 /var/lib/bxiie-cms

  log "Testing Apache configuration."
  /usr/sbin/apachectl configtest

  log "Reloading Apache."
  /bin/systemctl reload "$APACHE_SERVICE"

  log "Deployment complete."
done

# End of file.
EOF

sudo chmod 755 /srv/git/bxiie-cms.git/hooks/post-receive
```

### Important sudo note

The hook runs as the `deploy` user. The hook includes `chown`, `apachectl`, and `systemctl reload`, which normally require root.

You have two choices.

## Option A: Simpler first deploy, run hook tasks manually as root

For the first deployment, remove the `chown`, `apachectl`, and `systemctl` lines from the hook or expect them to fail. Then run the permission and reload commands manually after each push.

## Option B: Allow limited passwordless sudo for deploy

Recommended for repeatable deployment.

Create `/etc/sudoers.d/bxiie-deploy`:

```bash
sudo visudo -f /etc/sudoers.d/bxiie-deploy
```

Add:

```text
deploy ALL=(root) NOPASSWD: /usr/sbin/apachectl configtest, /bin/systemctl reload apache2, /usr/bin/chown -R deploy\:www-data /var/www/bxiie-cms, /usr/bin/chown -R www-data\:www-data /var/lib/bxiie-cms
```

Then update the hook commands:

```bash
sudo sed -i 's|chown -R deploy:www-data "$WORK_TREE"|sudo /usr/bin/chown -R deploy:www-data "$WORK_TREE"|' /srv/git/bxiie-cms.git/hooks/post-receive
sudo sed -i 's|chown -R www-data:www-data /var/lib/bxiie-cms|sudo /usr/bin/chown -R www-data:www-data /var/lib/bxiie-cms|' /srv/git/bxiie-cms.git/hooks/post-receive
sudo sed -i 's|/usr/sbin/apachectl configtest|sudo /usr/sbin/apachectl configtest|' /srv/git/bxiie-cms.git/hooks/post-receive
sudo sed -i 's|/bin/systemctl reload "$APACHE_SERVICE"|sudo /bin/systemctl reload "$APACHE_SERVICE"|' /srv/git/bxiie-cms.git/hooks/post-receive
```

Verify sudo permissions:

```bash
sudo -u deploy sudo /usr/sbin/apachectl configtest
sudo -u deploy sudo /bin/systemctl reload apache2
```

# Part 5: Configure Apache for the existing Bxiie vhost

## 1. Find the existing vhost

On the production server:

```bash
sudo apachectl -S
grep -R "ServerName bxiie.com\|ServerAlias.*bxiie.com\|DocumentRoot" /etc/apache2/sites-available /etc/apache2/sites-enabled
```

Write down the file that owns `bxiie.com`.

Likely locations:

```text
/etc/apache2/sites-available/bxiie.conf
/etc/apache2/sites-available/bxiie-le-ssl.conf
/etc/apache2/sites-enabled/bxiie.conf
/etc/apache2/sites-enabled/bxiie-le-ssl.conf
```

Edit the real file in `sites-available`, not the symlink in `sites-enabled`.

## 2. Update the HTTP vhost

Inside the port 80 vhost for `bxiie.com`, set:

```apache
DocumentRoot /var/www/bxiie-cms/public

<Directory /var/www/bxiie-cms/public>
    Options -Indexes +FollowSymLinks
    AllowOverride All
    Require all granted
</Directory>

SetEnv APP_NAME "Bxiie Artist CMS"
SetEnv BASE_URL "https://bxiie.com"
SetEnv DATABASE_PATH "/var/lib/bxiie-cms/database/bxiie.sqlite"
SetEnv STORAGE_PATH "/var/lib/bxiie-cms/storage"
SetEnv ADMIN_PATH "/admin"
SetEnv SESSION_NAME "bxiie_admin"
SetEnv DEFAULT_TENANT_SLUG "bxiie"
SetEnv DEFAULT_DOMAIN "bxiie.com"
SetEnv WATERMARK_TEXT "© Bxiie"
SetEnv MAIL_FROM "noreply@bxiie.com"
SetEnv CONTACT_TO "artist@bxiie.com"
```

If the HTTP vhost only redirects to HTTPS, it may not need all `SetEnv` lines. The HTTPS vhost definitely needs them.

## 3. Update the HTTPS vhost

Inside the port 443 Certbot-managed vhost for `bxiie.com`, set:

```apache
DocumentRoot /var/www/bxiie-cms/public

<Directory /var/www/bxiie-cms/public>
    Options -Indexes +FollowSymLinks
    AllowOverride All
    Require all granted
</Directory>

SetEnv APP_NAME "Bxiie Artist CMS"
SetEnv BASE_URL "https://bxiie.com"
SetEnv DATABASE_PATH "/var/lib/bxiie-cms/database/bxiie.sqlite"
SetEnv STORAGE_PATH "/var/lib/bxiie-cms/storage"
SetEnv ADMIN_PATH "/admin"
SetEnv SESSION_NAME "bxiie_admin"
SetEnv DEFAULT_TENANT_SLUG "bxiie"
SetEnv DEFAULT_DOMAIN "bxiie.com"
SetEnv WATERMARK_TEXT "© Bxiie"
SetEnv MAIL_FROM "noreply@bxiie.com"
SetEnv CONTACT_TO "artist@bxiie.com"
```

Preserve the existing Certbot lines, which will look similar to:

```apache
SSLCertificateFile /etc/letsencrypt/live/bxiie.com/fullchain.pem
SSLCertificateKeyFile /etc/letsencrypt/live/bxiie.com/privkey.pem
Include /etc/letsencrypt/options-ssl-apache.conf
```

Do not delete Certbot-managed SSL settings.

# Part 9: Backups

## 1. Manual database backup

```bash
sudo mkdir -p /var/backups/bxiie-cms
sudo cp /var/lib/bxiie-cms/database/bxiie.sqlite "/var/backups/bxiie-cms/bxiie.sqlite.$(date +%Y%m%d-%H%M%S).bak"
```

## 2. Manual storage backup

```bash
sudo tar -czf "/var/backups/bxiie-cms/storage.$(date +%Y%m%d-%H%M%S).tar.gz" -C /var/lib/bxiie-cms storage
```

## 3. Restore database from backup

Destructive step: this replaces the current database.

```bash
sudo systemctl stop apache2
sudo cp /var/backups/bxiie-cms/bxiie.sqlite.YYYYMMDD-HHMMSS.bak /var/lib/bxiie-cms/database/bxiie.sqlite
sudo chown www-data:www-data /var/lib/bxiie-cms/database/bxiie.sqlite
sudo systemctl start apache2
```

# Part 10: Rollback

## 1. Find previous commits

On the production server:

```bash
sudo -u deploy git --git-dir=/srv/git/bxiie-cms.git log --oneline --decorate --all -n 20
```

## 2. Roll back from workstation

On your development workstation:

```bash
cd ~/src/bxiie-cms
git log --oneline -n 20
git revert BAD_COMMIT_SHA
git push production main
```

Preferred rollback is `git revert`, because it preserves history and triggers the normal deployment hook.

## 3. Emergency checkout rollback on server

Only use this when the site is broken and you need immediate recovery.

```bash
sudo -u deploy git --git-dir=/srv/git/bxiie-cms.git --work-tree=/var/www/bxiie-cms checkout -f GOOD_COMMIT_SHA
sudo apachectl configtest
sudo systemctl reload apache2
```

# Part 11: Multitenant domain setup

## 1. DNS

Point the new artist domain to the same server IP address.

Example DNS records:

```text
A     exampleartist.com      YOUR_SERVER_IPV4
A     www.exampleartist.com  YOUR_SERVER_IPV4
```

## 2. Apache ServerAlias

Add the domain as a `ServerAlias` on the same vhost:

```apache
ServerAlias www.bxiie.com exampleartist.com www.exampleartist.com
```

## 3. Certificate

Once DNS resolves correctly, expand the existing certificate:

```bash
sudo certbot --apache -d bxiie.com -d www.bxiie.com -d exampleartist.com -d www.exampleartist.com
```

Only do this after the domain is real and DNS points to the server.

## 4. Tenant mapping

The app will need an admin function or database insert to map the new domain to the new tenant.

Example direct SQLite insert pattern, adjust IDs and slugs after the tenant-management UI exists:

```bash
sudo -u www-data sqlite3 /var/lib/bxiie-cms/database/bxiie.sqlite
```

Inside SQLite:

```sql
INSERT INTO tenants (slug, display_name, status, created_at)
VALUES ('exampleartist', 'Example Artist', 'active', datetime('now'));

INSERT INTO tenant_domains (tenant_id, domain)
SELECT id, 'exampleartist.com'
FROM tenants
WHERE slug = 'exampleartist';

INSERT INTO tenant_domains (tenant_id, domain)
SELECT id, 'www.exampleartist.com'
FROM tenants
WHERE slug = 'exampleartist';
```
